# 软工老师偏好 Skill

一个面向《软件工程基础》课程项目的 Codex Skill，用来把老师在课堂中反复强调的评分偏好转化为可复用的写作、审查和项目推进规范。

这个 Skill 适合在准备课程项目材料时使用，尤其是需求文档、产品原型说明、UML/DFD 设计文档、过程留痕材料、任务分解表、工具选型报告、测试与自动化计划，以及最终项目验收汇报。

## 核心定位

老师的评分重点不只是“系统最后能不能跑”，而是团队有没有真正走过软件工程过程：需求是否说清楚，原型是否可点击，设计是否能追溯到需求，任务是否分解到人，过程是否有证据，AI 和工具是否被合理使用并能解释清楚。

因此，这个 Skill 会优先帮助你检查：

- 是否有在线文档、会议记录、截图、签到、任务流转等过程证据
- 需求是否包含功能、性能、约束、验收标准和待确认问题
- 原型是否覆盖页面跳转、按钮状态、错误提示、空状态、加载状态和异常场景
- UML、DFD、接口、测试用例是否能和需求建立追踪关系
- 团队角色、任务分工、工作量估计和内部培训是否足够清晰
- AI 生成内容是否经过人工理解、审核和记录

## 适用场景

当你正在做以下事情时，可以调用这个 Skill：

- 起草或修改课程项目需求文档
- 审查甲方/乙方对接材料
- 设计产品原型说明或原型验收清单
- 梳理 UML、DFD、ER 图、接口文档和测试计划
- 准备组长评分表、任务分解表、考勤与培训记录
- 写最终验收汇报或项目总结
- 在提交前找出老师可能扣分的地方

## 使用方式

将本仓库放入 Codex 的 skills 目录，或把其中的 `SKILL.md` 复制到你的本地 Skill 目录：

```bash
~/.codex/skills/software-engineering-teacher-preferences/
```

之后可以在对话中这样使用：

```text
Use $software-engineering-teacher-preferences to review this requirements document.
```

也可以直接用中文说明任务：

```text
帮我按软工老师的偏好审查这份需求文档，指出扣分风险并给出补救动作。
```

## 设计原则

- 过程优先：把“做过什么”转化为可检查、可追溯的证据
- 需求前置：先把业务、边界、验收标准和变更规则说清楚
- 团队视角：强调角色分工、任务认领、工作量量化和内部培训
- 可解释 AI：鼓励使用 AI，但不能让团队变成看不懂产物的搬运工
- 提交前审查：每份材料都应该附带扣分风险和补救动作

## 备注

这个 Skill 来自课堂笔记的偏好归纳，适合服务课程项目提交与复盘。使用时请根据你们小组的真实材料补充证据链接，不要虚构签到、确认、评审或工具使用记录。
